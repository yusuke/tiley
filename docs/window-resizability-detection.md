# ウインドウリサイズ制約の検出方法

## 現在の検出アーキテクチャ

検出は2段階で行われる。

### 第1段階: プレビュー時の事前検出 (`detectResizability`)

グリッドオーバーレイでユーザーがドラッグ中に、対象ウインドウのリサイズ可否を検出する。
結果はPID単位でキャッシュされ、ターゲット切り替え時にクリアされる。

```
AccessibilityService.detectResizability(of:) → WindowResizability(.horizontal, .vertical)
```

#### 判定フロー (3段階)

```
1. kAXSizeAttribute が settable でない → .none (完全に非リサイズ)
2. AXFullScreenButton が存在する      → .both (完全にリサイズ可能)
3. それ以外                           → 1px probe (軸ごとに判定)
```

| 判定段階 | 方法 | 速度 | 破壊的か |
|---|---|---|---|
| 1. sizeSettable チェック | `AXUIElementIsAttributeSettable` | 即時 | No |
| 2. FullScreenButton チェック | `AXUIElementCopyAttributeValue` | 即時 | No |
| 3. 1px probe | サイズを±1px変更→読み戻し→復元 | 遅い | Yes (一瞬変更) |

### 第2段階: 移動後の事後検出 (`setFrame` の readback)

実際にウインドウを移動・リサイズした後、50ms待ってからサイズを読み戻し、
要求サイズと実際のサイズに差異があれば `constrained = true` を返す。
ステータスメッセージに「(size adjusted due to window constraints)」と表示される。

## 検出できるパターン

### 完全に非リサイズのウインドウ
- **例**: 計算機 (Calculator)
- **判定**: ステップ1で `sizeSettable = false` → `.none`
- **精度**: 確実
- **プレビュー**: 全体が赤く表示される

### 完全にリサイズ可能なウインドウ
- **例**: Finder, Safari, Xcode, ターミナル, メモ, メール, Slack, IntelliJ IDEA, Docker Desktop, LINE
- **判定**: ステップ2で `AXFullScreenButton` が存在 → `.both`
- **精度**: 確実 (テスト済み: 全て `hasFSButton=true`)
- **プレビュー**: 通常表示 (青色のみ)

### 一部の軸が固定されたウインドウ
- **例**: システム設定 (横幅固定、縦はリサイズ可能)
- **判定**: ステップ3の1px probeで軸ごとに判定
- **精度**: 固定か否かは正確に検出可能
- **プレビュー**: 固定軸方向のはみ出し部分が赤く表示される

## 検出できないパターン

### 最小幅・最大幅 (min/max の範囲制約)
- **例**: 特定の最小幅 (例: 400px以上) を持つウインドウ
- **現状**: 1px probeは「固定か否か」のみを判定。最小幅が現在の幅より小さければ、+1pxの変更は成功するため「リサイズ可能」と判定される。しかし、選択した領域が最小幅より狭い場合、実際にはそこまで縮小できない。
- **影響**: プレビューでは青く表示されるが、実際に移動すると最小幅に制約される。事後検出 (setFrame readback) で `constrained` フラグは立つが、事前のプレビューには反映されない。

### 最大サイズ制約
- **例**: 特定の最大サイズを持つウインドウ
- **現状**: 同上。+1pxの変更が最大値以下なら「リサイズ可能」と判定されるが、選択領域が最大値を超える場合は実際には拡大できない。
- **影響**: プレビューには反映されず、事後検出のみ。

### AXFullScreenButton があるが部分的に制約があるウインドウ
- **例**: (現時点では未確認だが理論上あり得る) フルスクリーンボタンがあるが、特定の軸のリサイズが制約されているウインドウ
- **現状**: ステップ2で `.both` と即時判定され、1px probeがスキップされる。
- **影響**: 制約があっても「完全にリサイズ可能」として表示される。

### Electron系アプリの独自サイズ制約
- **例**: 一部のElectronアプリは最小/最大サイズをアプリ内部で制御
- **現状**: AX APIレベルではsettableと報告されるが、実際の制約はアプリのレンダラープロセスが管理
- **影響**: 事前検出では拾えない可能性がある。事後検出は有効。

### ウインドウの現在サイズがちょうど最小値/最大値の場合
- **例**: ウインドウが既に最小幅の状態
- **現状**: +1px probeで縮小方向は試さないため、縮小不可であることは検出しない。ただし、現在の判定は「固定か否か」のみなので、+1pxが成功すれば「リサイズ可能」と判定される。
- **影響**: 最小値に張り付いた状態でも「リサイズ可能」と判定される (拡大方向には確かにリサイズ可能なので、誤判定とは言い切れない)。

### ダイアログ・シートなどの特殊ウインドウ
- **例**: 保存ダイアログ、警告シートなど
- **現状**: subrole が `AXDialog` などの場合の特別な処理はない。`sizeSettable` の結果に依存。
- **影響**: 大半のダイアログは `sizeSettable=false` で正しく `.none` と判定されるはず。

## 実行タイミングと性能

| 処理 | 実行タイミング | 対象 | 性能影響 |
|---|---|---|---|
| `detectResizability` | プレビュー表示時 (遅延実行) | ターゲットウインドウのみ | 小さい (1ウインドウ) |
| `setFrame` readback | ウインドウ移動後 | ターゲットウインドウのみ | 50ms待機あり |
| ウインドウ列挙 (`allWindowTargets`) | Tab切り替え時 | 全ウインドウ | **probeなし** (高速) |

以前の実装ではウインドウ列挙時に全ウインドウに対して1px probeを行っていたが、
現在はターゲットウインドウに対してのみ、かつプレビュー表示時に遅延実行するように改善済み。
さらに、大多数のウインドウは `AXFullScreenButton` の有無で即時判定されるため、
1px probeが発生するのは `AXFullScreenButton` がない少数のウインドウに限られる。

| Safari | true | あり | 完全にリサイズ可能 |
| Xcode | true | あり | 完全にリサイズ可能 |
| ターミナル | true | あり | 完全にリサイズ可能 |
| Docker Desktop | true | あり | 完全にリサイズ可能 |
| Slack | true | あり | 完全にリサイズ可能 |
| IntelliJ IDEA | true | あり | 完全にリサイズ可能 |
| LINE | true | あり | 完全にリサイズ可能 |
| プレビュー | true | あり | 完全にリサイズ可能 |
| メール | true | あり | 完全にリサイズ可能 |
| メモ | true | あり | 完全にリサイズ可能 |
| カレンダー | true | あり | 完全にリサイズ可能 |
| システム設定 | true | **なし** | 横幅固定、縦リサイズ可 |
| 計算機 | **false** | なし | 完全に非リサイズ |

### 調査で判明した使えないAX属性

| 属性 | 結果 |
|---|---|
| `kAXGrowAreaAttribute` | 全アプリで `ABSENT` (code -25200)。macOS 15時点では使用不可 |
| `kAXMinimumSize` / `kAXMaximumSize` | エラー -25205。macOS 15のほとんどのアプリで取得不可 |
| `AXUIElementIsAttributeSettable(kAXSizeAttribute)` | システム設定で `true` を返すが実際にはサイズ変更を無視 (信頼性不足) |
